# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A version-controlled Ghostty terminal configuration for macOS and Linux. It is not an application — it is a set of `.ghostty` config files, a custom theme, shell prompt snippets, and Bash installer scripts. There is no build step and no compiled code.

## Commands

```bash
make install         # symlink config/ into $XDG_CONFIG_HOME/ghostty (auto-detects platform)
make install-copy    # same, but copies files instead of symlinking
make install-prompt  # add the git-aware prompt block to ~/.bashrc or ~/.zshrc
make validate        # ghostty +validate-config against config/config.ghostty
make doctor          # environment/config diagnostics
make backup          # back up existing Ghostty config
make check           # bash -n syntax check of all scripts and bash_prompt.sh
make tree            # list tracked files

# Switch the active platform overlay after install (no reinstall needed):
./scripts/use-profile.sh macos|linux|minimal

# Most scripts support --dry-run to preview actions without touching the filesystem:
./scripts/install.sh --dry-run
```

There is no test suite; `make check` (a `bash -n` parse check) is the closest thing to a lint/test gate. Run it after editing any script. Note that `make validate` and `make doctor` require the `ghostty` CLI on `PATH` and otherwise exit early.

## Config layering (the core architecture)

`config/config.ghostty` is the single shared base loaded on every platform. It composes two optional layers via Ghostty's `config-file = ?<name>` syntax (the `?` means "load if present, ignore if missing"):

1. `profile.ghostty` — the platform overlay. This is a symlink (or copy) created by the installer pointing to one of `config/profiles/{macos,linux,minimal}.ghostty`. It is regenerated by `install.sh` and can be repointed by `use-profile.sh`.
2. `local.ghostty` — untracked, machine-specific private overrides. Never committed. Users copy `local.example.ghostty` to create it.

Precedence: base → profile → local (later `config-file` includes win). **Per-machine tweaks belong in `local.ghostty`, never in `config.ghostty`.** When changing behavior, decide which layer it belongs to: universal defaults go in the base, OS-specific behavior in a profile, and anything personal stays local.

## Installer behavior

`scripts/install.sh` targets `${XDG_CONFIG_HOME:-$HOME/.config}/ghostty` on both platforms (the XDG path is used on macOS too, deliberately, to keep both OSes consistent). It backs up any existing config dir to `<dir>.backup-<timestamp>` before replacing it, then either symlinks (default) or copies `config/`, and finally writes the `profile.ghostty` link for the detected/forced profile.

## Conventions

- Ghostty config comments must be on their own line. Do **not** add trailing inline comments in `.ghostty` files — Ghostty does not parse them.
- The prompt scripts (`config/shell/bash_prompt.sh`, `zsh_prompt.zsh`) are dependency-free and self-contained; `starship.toml` is the optional richer alternative. The installer inserts/locates the block by the marker string `ghostty muted coding prompt` — keep that marker intact if editing prompt install logic.
- Scripts use `set -euo pipefail` and validate the profile argument against `macos|linux|minimal`. Match this style in new scripts.
